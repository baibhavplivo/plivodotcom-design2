<p><em>Plivo’s CTO Michael Ricordeau writes …</em></p><p>I’ve been exploring AI and wanted to test a few things on my own because that’s what I enjoy doing as <a href="https://www.linkedin.com/in/michaelricordeau/">CTO at Plivo</a>. I wanted to learn about <a href="https://openai.com/">OpenAI</a> tools and build something useful for the company at the same time. The result was our new /askplivo chatbot for Slack.</p><p>We have a lot of resources on our website, including API documentation, blog posts, and use case guides. It can be challenging for our employees to navigate through them and find what they want.</p><p>Occasionally, I would see questions popping up on our Slack channels about a feature, or telecom regulations (I’m looking at you, <a href="https://www.plivo.com/docs/sms/a2p-10dlc/quickstart/">10DLC</a>), or country coverage. If we could have a Q&amp;A chatbot assistant capable of answering questions instead of waiting (desperately, sometimes) for a co-worker to reply we could be more productive.</p><p>For instance, imagine you’re a Plivo support engineer and a customer is asking whether Plivo supports <a href="https://plivo-webflow.webflow.io/sms">SMS</a> in France. (I use France as an example because that’s where I’m from, but you can use any other country; <a href="https://plivo-webflow.webflow.io/sms-coverage/rs">we support more than 190</a>.) With a chatbot, you could get an answer in less than a minute.</p><p>Because my goal was to learn more about AI and OpenAI, I started exploring how I could build a chatbot that used Plivo’s content enhanced with the power of AI. I wanted something that would be smart enough to answer with relevant information and not <a href="https://en.wikipedia.org/wiki/Hallucination_(artificial_intelligence)">hallucinate</a>, which is a well-known problem.</p><h2>Designing the AI chatbot</h2><p>I like Python for prototyping and <a href="https://en.wikipedia.org/wiki/Minimum_viable_product">MVP</a> because the syntax is simple, and I’ve been using it for 20 years. Granted, it’s not super performant, but I’m not planning to build the next Google here. I wanted a lightweight web framework, so I used <a href="https://flask.palletsprojects.com/">Flask</a> to expose the API to interact with Slack and <a href="https://python-rq.org/">RQ</a> to process jobs in the background.</p><p>For AI, I found LangChain, a Python (and TypeScript) framework that abstracts many of the <a href="https://en.wikipedia.org/wiki/Large_language_model#:~:text=A">LLM</a> and indexing parts. It’s well-integrated with the OpenAI API.</p><p>To store, index, and query (search) the data, I wanted to use a <a href="https://learn.microsoft.com/en-us/semantic-kernel/memories/vector-db">vector database</a>. I had many challenges choosing one: I tried <a href="https://redis.io/docs/stack/search/">RedisSearch</a> and <a href="https://github.com/facebookresearch/faiss">FAISS</a> but ended up using <a href="https://qdrant.tech/">Qdrant</a>; the cloud version offers a free tier that was good enough for my use case, and it’s written in Rust, so it’s memory-efficient.</p><p>Finally, I picked <a href="https://fly.io/">Fly</a> to host the software. I wanted to test Fly because it’s cheap and easy to deploy and scale with a Dockerfile, and I didn’t want to use Heroku anymore.</p><p>Here’s how everything fits together:</p><h2>Technical details and pain points</h2><p>Building the application was both fun and a learning experience.</p><p>I had high hopes for using FAISS, but it’s not compatible between x86_64 and arm64 architectures, so if, like me, you’re developing on an arm64-based Apple M1/M2, you cannot build your FAISS index file and upload it to your x86_64 server — and the Fly hosting platform doesn’t support arm64. I turned instead to Qdrant.</p><p>To feed the data in the Qdrant vector database, I took the <a href="https://www.plivo.com/">Plivo</a> website sitemap.xml file, scraped all of the URLs, extracted the content, and transformed it into LLM embeddings, then inserted it in Qdrant. If you do this, be warned that OpenAI will bill you for the embeddings.</p><p>Next, I loaded the documents. LangChain, by default, tries to load all documents in memory, so I got quite a lot of OOM killer in my Fly instance. I didn’t want to triple the memory usage (and my bill), so I built a data importer to process documents in chunks instead of loading everything in one shot. Also, LangChain has no GitHub data importer, so I wrote one.</p><p>At this point, I had the resources I needed for the chatbot hosted on Fly. I should note that while most of the tools I used are lightweight and fast enough for our use case, OpenAI can be a bottleneck. If you use the GPT-4 model, response time can take up to two minutes, depending on the complexity of your question. I had to change the API timeout for the OpenAI Python SDK to 180 seconds.</p><p>Once I had the back end set up, the next step was to create the Slack bot app from the Slack admin console. I’ve documented the process of configuring and deploying the application in the README page of the project’s GitHub repository.</p><h2>Why don’t we consult the oracle?</h2><p>After a few hours of development time, I had a working chatbot. Here’s what using it looks like.</p><h2>Et voilà!</h2><p>Want to develop something like the /askplivo bot? Check out <a href="https://github.com/plivo/plivoaskme">my code</a> — it’s open source, so feel free to use it and fork it. You’ll find there’s room for improvement and a few challenges to overcome:</p><ul><li>The search query is optimized for text. Most of the time, it won’t use the GitHub indexed code; I need to dig into that.</li><li>It occasionally hallucinates responses — making up a 10DLC API resource, for instance. I’m planning to experiment with <a href="https://github.com/dioptra-ai/anchor-gpt">Anchor GPT</a> to fix that.</li><li>If you add new documents, you have to re-index all documents from scratch, which is slow and costly.</li><li>There is no chat session concept — each question/response is always unrelated to the previous ones. I’m not sure we need this feature.</li></ul><p>If you use the code, and especially if you enhance it in a way it can benefit others, feel free to contribute and send a PR.<br></p>
